1 · Visión y alcance

Motor de renderizado HTML/CSS→PDF 100% PHP: sin binarios externos, sin Node, sin navegador headless. Instalable con composer require en cualquier hosting. Ambición final: renderizar páginas Bootstrap y Tailwind reales. El desarrollo es por hitos de capacidad CSS: cada milestone deja el motor útil, testeado y publicable.

  • Política de dependencias: PHP 8.4+ y extensiones estándar (dom, mbstring; gd/imagick opcionales para imágenes). Una sola dependencia Composer inicial: sabberworm/php-css-parser, aislada tras interfaz y sustituible.
  • Distribución: core framework-agnostic; adapter Laravel (facade, service provider, ->view()) como package separado tras M2. Desarrollo y pruebas con un index.php playground + CLI.
  • Tipografía v1: escrituras latinas con kerning. Shaping complejo (árabe, índico) explícitamente fuera hasta tener motor estable.
  • Sin JavaScript: el motor renderiza documentos, no ejecuta scripts. Nunca en alcance.

2 · Decisiones clave — las cuatro apuestas

Derivadas de primeros principios (un PDF no tiene layout: solo posiciones absolutas; lo irreducible es medir y fragmentar) y de las capacidades diferenciales de PHP frente a los incumbentes (dompdf, WeasyPrint):

1CSS compilado a PHP, no interpretado

En servidor, el stylesheet es estático entre deploys. Un compilador convierte CSS en un matcher PHP plano (selectores como condicionales generados, especificidad pre-resuelta) que opcache mantiene caliente — el modelo Blade aplicado a CSS. Bootstrap se distribuye ya precompilado dentro del package. Modo interpretado para desarrollo. La interfaz StyleSource asume ambos modos desde el día 1.

2Streaming de páginas con generators

El paginador es un generator (yield Page): cada página terminada se pinta, se serializa al PDF y se libera. Memoria O(página), no O(documento). Restricción asumida: no se reabre una página emitida; los contadores hacia delante (página X de Y) usan placeholders parcheados al cierre del fichero.

3Kernel mínimo + layouts como plugins

El kernel solo conoce la interfaz FormattingContext (medir tamaños intrínsecos, colocar hijos, fragmentar). Block flow, inline, tablas, flex, grid: cada uno es un módulo independiente que implementa esa interfaz. Es API pública de extensión y el mapa natural para desarrollo subagent-driven con un agente por subsistema.

4IR lista para plantillas (v2, solo diseño ahora)

La representación intermedia entre layout y paint (display list de primitivas posicionadas) se diseña separable en parte estática + huecos dinámicos, para que una v2 pueda pre-compilar el layout de plantillas repetitivas. En v1 no se implementa: solo se evita cerrarle la puerta.

Donde deliberadamente NO se innova

Los algoritmos de layout siguen las specs W3C al pie de la letra (line breaking, flex resolution, cascade). Las specs garantizan terminación, predictibilidad y el resultado correcto contra el que todos comparan. La innovación está en la arquitectura que los rodea.

API pública mínima v1

$engine = Engine::make()
    ->stylesheet('app.css')          // o ->bootstrap() o ->tailwind()
    ->paper('a4', margins: '2cm');

$engine->render($html)->save('out.pdf');   // o ->stream() para HTTP

6 · Estándares de código y disciplina de acoplamiento

Requisito de primer orden: si un dev clona el repo, debe encontrar el código exquisito — sencillo de leer, con abstracciones sólidas y acoplamiento mínimo verificado por CI.

  • PER-CS 2.0 + PSR-4, declare(strict_types=1) universal, PHPStan nivel máximo y php-cs-fixer en CI desde el commit 1.
  • Inmutabilidad por defecto: ComputedStyle, geometrías, fragmentos y primitivas de paint son value objects final readonly. Los datos fluyen hacia delante; ninguna etapa muta lo que produjo la anterior.
  • Dependencias explícitas por constructor, cero contenedor DI en el core (el container llega en el adapter Laravel).
  • Regla de dependencia verificada con deptrac en CI: el acoplamiento bajo es un test que falla, no una intención.
  • Código = spec anotada: cada algoritmo referencia en docblock la sección W3C que implementa y usa el vocabulario de la spec (hypotheticalMainSize, no $tmp).

Flechas = flujo de datos por el pipeline. Las dependencias de código solo pueden seguir estas mismas aristas (el consumidor depende del productor, siempre contra interfaces, nunca clases concretas). Layout/ no sabe que Pdf/ existe; Css/ no sabe qué es una caja. deptrac falla el build ante cualquier arista nueva.

7 · Manejo de errores

SituaciónComportamiento
CSS desconocido/inválidoSe descarta la declaración y sigue (per spec). Cada descarte queda en RenderReport (warnings estructurados). Modo strict(): excepción al primer descarte.
HTML rotoLo absorbe el parser HTML5 por spec. Nunca es error.
Fuente/glifo ausenteCadena de fallback declarativa; glifo de sustitución + warning. Nunca excepción a mitad de página.
HTML patológicoLímites configurables maxPages / maxRenderTime; excepción tipada al superarlos.
Errores internosExcepciones propias (LayoutException, FontException…) con contexto del nodo causante.

8 · Estrategia de testing

  1. Unitaria por subsistema (Pest): cada FormattingContext se testea con árboles de cajas sintéticos, sin HTML ni PDF de por medio.
  2. Golden files de geometría: corpus HTML → volcado JSON del árbol de fragmentos comparado contra snapshot. Los bugs de layout son diffs de números.
  3. Chrome como oráculo (diferencial, solo CI): mismo corpus por Chrome print-to-PDF; se extraen posiciones de texto de ambos PDFs y se comparan con tolerancia. Chrome no es dependencia del motor.
  4. Validez y rendimiento: qpdf/verapdf sobre cada PDF generado + benchmarks de memoria que garantizan el O(página) del streaming.

10 · Fuera de alcance v1

  • Ejecución de JavaScript (nunca en alcance).
  • Shaping de escrituras complejas (árabe, índico) y bidi completo.
  • Pre-compilación de layout de plantillas (v2; la IR queda preparada).
  • SVG completo (subset básico se evaluará en M8), animaciones/transiciones, media queries distintas de print.
  • Accesibilidad PDF/UA y PDF/A (candidatas post-v1; el modelo de objetos no las bloquea).

11 · Riesgos y mitigaciones

RiesgoMitigación
Rendimiento de PHP en layout de documentos grandesBenchmarks desde M0 en CI; value objects ligeros; streaming ya reduce memoria; opcache+JIT; los documentos tienen miles de cajas, no millones.
Line breaking / inline es el subsistema más difícilM1 dedicado en exclusiva; UAX#14 tiene suite de tests oficial de Unicode reutilizable como fixture.
sabberworm se queda corto para el compiladorAislado tras interfaz en Css/; sustituible por tokenizer propio sin tocar otros subsistemas.
Scope creep hacia "ser un navegador"Cada propiedad CSS entra solo por milestone declarado; lo no soportado se ignora con warning en RenderReport (nunca a medias).
Divergencia visual respecto a navegadoresChrome como oráculo en CI desde M0: la divergencia se mide, no se opina.

12 · Referencias

  • Specs W3C: CSS 2.2, css-inline-3, css-flexbox-1, css-break-3, css-page-3, css-tables-3.
  • Unicode UAX#14 (line breaking) — incluye test suite oficial.
  • PDF: ISO 32000-1 (PDF 1.7).
  • Prior art: WeasyPrint (arquitectura de referencia, Python), dompdf (font-lib y corpus de tests), typeset.sh (prueba de viabilidad comercial en PHP), LayoutNG de Chrome (inspiración del kernel+contexts).
  • Parser HTML5 nativo: \Dom\HTMLDocument (PHP 8.4, RFC "New DOM API").
